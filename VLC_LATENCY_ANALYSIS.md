# VLC 플레이어 레이턴시 정밀 분석 보고서

## 1. 개요

VLC 플레이어로 치지직 HLS 라이브 스트림 재생 시 발생하는 레이턴시(지연)를 단계별로 분석하고, 각 구간의 개선안을 도출합니다.

**목표:** 치지직 웹사이트 수준의 라이브 동기화 달성 (체감 지연 3~5초 이내)

---

## 2. 현재 레이턴시 구조 분석

### 2.1 재생 시작부터 화면 출력까지의 파이프라인

```
[사용자가 채널 클릭]
  ↓  ① API 호출 (~200ms)
[치지직 API → HLS 마스터 매니페스트 URL 획득]
  ↓  ② CDN 워밍 HEAD 요청 (~100-300ms)
[CDN TCP/TLS 핸드셰이크 완료]
  ↓  ③ 마스터 매니페스트 다운로드 + 파싱 (~100-200ms)
[1080p variant URL 해석]
  ↓  ④ VLCMedia 생성 + 옵션 설정 (~10ms)
  ↓  ⑤ VLC play() 호출
[VLC 내부: HTTP 클라이언트 → variant URL 접속]
  ↓  ⑥ VLC 미디어 플레이리스트 로드 (~200-500ms)
[HLS 청크리스트 파싱, 세그먼트 목록 획득]
  ↓  ⑦ adaptive-livedelay 적용 → 엣지에서 6초 뒤 세그먼트 선택
  ↓  ⑧ 첫 세그먼트 다운로드 (~500-1500ms, 세그먼트 크기 의존)
[fMP4/TS 세그먼트 수신]
  ↓  ⑨ 디먹서 + 키프레임 탐색 (~50-200ms)
  ↓  ⑩ VideoToolbox 디코더 초기화 (~100-300ms)
  ↓  ⑪ 첫 프레임 디코딩 + 렌더링
[화면에 비디오 출력] ← TTFP (Time To First Picture)
```

### 2.2 현재 레이턴시 분포 추정

| 구간 | 소요 시간 | 설명 |
|------|----------|------|
| ① API 호출 | ~200ms | 치지직 liveDetail API |
| ② CDN 워밍 | ~100-300ms | HEAD 요청 (TCP+TLS 선행) |
| ③ 매니페스트 파싱 | ~100-200ms | 마스터 M3U8 다운로드+파싱 |
| ④⑤ VLC 초기화 | ~10-50ms | VLCMedia 생성, play() 호출 |
| ⑥ 플레이리스트 로드 | ~200-500ms | VLC 내부 HTTP → chunklist 파싱 |
| ⑦ 라이브 엣지 거리 | **6,000ms** | `adaptive-livedelay=6` |
| ⑧ 세그먼트 다운로드 | ~500-1,500ms | 2-4초 세그먼트, 네트워크 의존 |
| ⑨ 디먹서+키프레임 | ~50-200ms | fMP4 박스 파싱, IDR 탐색 |
| ⑩ 디코더 초기화 | ~100-300ms | VideoToolbox 세션 생성 |
| ⑪ 첫 프레임 렌더 | ~16-33ms | 60fps=16ms, 30fps=33ms |
| **합계 (TTFP)** | **~2-4초** | CDN 워밍 포함 |
| **라이브 지연** | **~8-12초** | TTFP + 라이브 엣지 거리 + 버퍼링 |

### 2.3 지속적 재생 중 레이턴시 구성

```
[라이브 생방송 시점] ─────────────────────────────────┐
                                                       │
    adaptive-livedelay (6초)                           │ 총 라이브 지연
    ├── CDN 세그먼트 가용 대기                          │ ~8-12초
    │                                                  │
    network-caching (3.5초)                            │
    ├── 세그먼트 다운로드 → 디먹서 버퍼                   │
    │                                                  │
    live-caching (2.5초)                               │
    ├── 디코더 → 렌더러 파이프라인 버퍼                   │
    │                                                  │
    clock-jitter (8초 허용)                             │
    ├── PTS 편차 흡수 (실제 추가 지연은 아님)             │
                                                       │
[사용자 화면 출력] ←──────────────────────────────────────┘
```

## 3. 현재 설정의 레이턴시 기여도 분석

### 3.1 `adaptive-livedelay=6` (최대 기여 요인)

**현재 설정:** 라이브 엣지에서 6초 뒤에서 재생
**기여 레이턴시:** ~6초 (가장 큰 비중)

| 값 | 장점 | 단점 |
|---|------|------|
| 3초 | 라이브 동기화 우수 | 세그먼트 404/timeout 위험 |
| 4초 | 적당한 균형 | CDN 지연 시 버퍼링 가능성 |
| **6초 (현재)** | 매우 안정적 | **라이브 체감 지연 큼** |
| 8초 | 극단적 안정 | 라이브 의미 없음 |

**분석:** 치지직 CDN은 보통 2-4초 세그먼트를 사용하며, CDN 엣지 서버 에서 최신 세그먼트 가용까지 1-2초 소요. `adaptive-livedelay=6`은 과도하게 보수적.

### 3.2 `network-caching=3500` + `live-caching=2500`

**기여 레이턴시:** ~3.5초 + ~2.5초 = 최대 ~6초 (중복 계산됨, 실제 ~3.5초)

VLC의 캐싱 메커니즘:
- `network-caching`: 네트워크에서 수신한 데이터를 디먹서에 전달하기 전에 버퍼링하는 시간
- `live-caching`: 라이브 스트림 전용 추가 버퍼 (network-caching과 **병렬이 아닌 대체** 관계)
- 실제 VLC 동작: `max(network-caching, live-caching)` → 실질 버퍼 ~3.5초

**분석:** VLC 소스코드 분석 결과, 라이브 스트림에서는 `live-caching`이 우선 적용되고 `network-caching`은 보조적으로 작용. 실질 추가 지연은 `live-caching` (~2.5초).

### 3.3 `clock-jitter=8000`

**실질 레이턴시 기여:** 0ms (지터 허용 범위일 뿐, 추가 버퍼링 아님)

VLC의 clock-jitter는 PTS 타임스탬프 편차를 "허용"하는 범위. 라이브에서 패킷이 일시적으로 지연 도착해도 프레임을 드롭하지 않고 재생 연속성을 유지.

**분석:** 8000ms는 안정성에 기여하며 레이턴시에는 영향 없음. 현재 설정 적절.

### 3.4 `cr-average=40`

**실질 레이턴시 기여:** 간접적 (~100-200ms)

PCR(Program Clock Reference)의 이동 평균 윈도우. 높을수록 클럭 변동을 더 많이 평활화하나, 새로운 클럭 값 반영이 느려짐.

**분석:** 40은 안정성 위주. 레이턴시 민감 모드에서는 20-30으로 줄일 수 있음.

### 3.5 `adaptive-maxbuffer=12`

**실질 레이턴시 기여:** 간접적 (프리페치 범위, 직접 지연 아님)

최대 12초치 세그먼트를 미리 다운로드 가능. 버퍼 고갈 방지용이며 직접적인 레이턴시 증가는 아님.

**분석:** 현재 설정 적절. 러페치 범위가 커도 재생은 라이브 엣지 기준이므로 지연 무관.

## 4. 치지직 웹사이트와의 비교

| 항목 | 치지직 웹 (hls.js) | CView VLC | 차이 |
|------|-------------------|-----------|------|
| HLS 라이브러리 | hls.js (JS) | VLC adaptive demux | - |
| LL-HLS 지원 | `EXT-X-PART` 사용 | 미지원 | **핵심 차이** |
| 라이브 엣지 거리 | ~2-3초 | **6초** | **+3-4초** |
| 네트워크 버퍼 | ~1-2초 | ~3.5초 | **+1.5-2.5초** |
| 디코더 | MSE (브라우저 내장) | VideoToolbox | 동등 |
| 총 라이브 지연 | **3-5초** | **8-12초** | **+5-7초** |

### 4.1 핵심 차이점

**① LL-HLS (Low-Latency HLS)**
치지직 웹은 `EXT-X-PART`(부분 세그먼트)와 `EXT-X-PRELOAD-HINT`를 활용하여 세그먼트 완성 전에 부분 데이터를 수신. VLC의 adaptive demux 모듈은 이 기능을 **지원하지 않음**.

**② 라이브 엣지 거리**
hls.js는 라이브 엣지에서 2-3 세그먼트 뒤에서 시작하지만, VLC의 `adaptive-livedelay=6`은 6초 뒤. 이것이 가장 큰 체감 차이.

**③ ABR 전략**
hls.js는 세그먼트 단위 대역폭 측정으로 빠른 ABR 전환. VLC ABR은 현재 비활성 (1080p variant URL 직접 전달).

## 5. 개선안

### 🟢 개선안 1: `adaptive-livedelay` 축소 (6→3초)

**예상 효과: 레이턴시 -3초 (가장 큰 개선)**
**위험도: 중**

```
현재: adaptive-livedelay=6 → 라이브 6초 뒤
개선: adaptive-livedelay=3 → 라이브 3초 뒤
```

**구현 방법:**
- `configureMediaOptions()`에서 `adaptive-livedelay` 값을 프로파일별로 분리
- `lowLatency` 프로파일: `adaptive-livedelay=2` (공격적)
- `normal` 프로파일: `adaptive-livedelay=3` (균형) 
- `highBuffer` 프로파일: `adaptive-livedelay=5` (보수적)

**위험 완화:**
- 세그먼트 404 발생 시 자동 1~2초 후퇴 (adaptive fallback)
- 연속 404가 3회 이상이면 adaptive-livedelay를 동적으로 증가

**구현 위치:** `VLCPlayerEngine.swift` `configureMediaOptions()`

---

### 🟢 개선안 2: 라이브 캐싱 프로파일 최적화

**예상 효과: 레이턴시 -1~2초**
**위험도: 낮음**

```
현재 (normal):  network-caching=3500, live-caching=2500
개선 (normal):  network-caching=2500, live-caching=2000
개선 (lowLatency): network-caching=1500, live-caching=1200
```

| 프로파일 | network-caching | live-caching | 예상 지연 |
|---------|----------------|-------------|----------|
| lowLatency (개선) | 1500ms | 1200ms | **~3-4초** |
| normal (개선) | 2500ms | 2000ms | **~5-7초** |
| highBuffer | 5000ms | 3000ms | ~10-12초 |

**구현 위치:** `VLCPlayerEngine.swift` `VLCStreamingProfile` enum

---

### 🟢 개선안 3: CDN 워밍 + 매니페스트 파싱 병렬화

**예상 효과: TTFP -200~500ms**
**위험도: 낮음**

현재:
```
CDN 워밍 (HEAD) → 매니페스트 다운로드 → 파싱 → VLC play()
(순차 실행)
```

개선:
```
┌── CDN 워밍 (HEAD)
├── 매니페스트 다운로드 + 파싱
└── (병렬 완료 대기)
     ↓
VLC play()
```

**구현 위치:** `StreamCoordinator.swift` `startStream()`

---

### 🟡 개선안 4: `cr-average` 저지연 모드 분리

**예상 효과: 레이턴시 -100~200ms**
**위험도: 낮음**

```
현재: 모든 프로파일 cr-average=40
개선:
  - lowLatency: cr-average=20 (빠른 클럭 추적)
  - normal: cr-average=30 (적당한 평활화)
  - highBuffer: cr-average=40 (최대 안정성)
```

**구현 위치:** `VLCPlayerEngine.swift` `configureMediaOptions()`

---

### 🟡 개선안 5: 매니페스트 갱신 주기 최적화

**예상 효과: 복구 레이턴시 -5~15초**
**위험도: 낮음**

```
현재: 30초 주기 매니페스트 갱신
개선:
  - lowLatency: 15초 주기
  - normal: 20초 주기
  - highBuffer: 30초 주기
```

**구현 위치:** `StreamCoordinator.swift` `startManifestRefreshTimer()`

---

### 🟡 개선안 6: 기본 프로파일 전환 전략

**예상 효과: 체감 레이턴시 -3~5초**
**위험도: 중**

현재 기본 프로파일이 `normal`이지만, 대부분의 라이브 시청은 저지연이 선호됨.

**제안:**
- 기본 프로파일을 `lowLatency`로 변경
- 네트워크 불안정(healthScore < 0.5 지속) 시에만 자동으로 `normal` → `highBuffer` 전환
- 사용자 설정에 "저지연 모드" 토글 추가

**구현 위치:** `VLCPlayerEngine.swift` `streamingProfile` 기본값

---

### 🔵 개선안 7: VLC 내부 타임스탬프 기반 라이브 동기화

**예상 효과: 장시간 재생 시 drift 방지**
**위험도: 중**

장시간 라이브 시청 시 VLC 내부 클럭이 점진적으로 drift하여 라이브 엣지와의 거리가 증가할 수 있음.

**제안:**
- 주기적(60초)으로 VLC의 `player.time` vs HLS 매니페스트의 최신 세그먼트 타임스탬프 비교
- drift > 5초이면 `player.time` 점프 또는 재연결로 라이브 엣지 재동기화
- 재생 중도 아닌 자연 drift이므로 무중단 동기화 가능

**구현 위치:** `VLCPlayerEngine.swift` 새 메서드 `checkLiveSyncDrift()`

---

### 🔵 개선안 8: 세그먼트 프리페치 최적화

**예상 효과: 세그먼트 전환 갭 감소 -100~300ms** 
**위험도: 낮음**

VLC의 `adaptive-maxbuffer=12`는 프리페치 범위가 넓지만, 첫 세그먼트 로드 시 지연이 있을 수 있음.

**제안:**
- `adaptive-maxbuffer`를 프로파일별로 분리
  - lowLatency: `adaptive-maxbuffer=6` (필요 최소 프리페치)
  - normal: `adaptive-maxbuffer=8`
  - highBuffer: `adaptive-maxbuffer=12`

**구현 위치:** `VLCPlayerEngine.swift` `configureMediaOptions()`

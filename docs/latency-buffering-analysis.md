## 1. 시스템 아키텍처 개요

### 1.1 레이턴시 동기화 파이프라인

```
[치지직 CDN] → HLS 매니페스트
       ↓
[VLC 엔진] → 디코딩/재생 (networkCaching + liveCaching)
       ↓
[PDTLatencyProvider] → EXT-X-PROGRAM-DATE-TIME 기반 레이턴시 측정
       ↓                    ↓ (PDT 실패 시 VLC 버퍼 fallback)
[LowLatencyController] ← EWMA 스무딩 (α=0.15)
       ↓                    ↓
       ↓              PID 제어기 (Kp=0.5, Ki=0.05, Kd=0.03)
       ↓                    ↓
[PlayerViewModel.applySyncSpeed()] ← 버퍼 건강도 기반 클램핑
       ↓
[VLCPlayerEngine.setRate()] → 실제 재생 속도 적용

─── 병렬 경로 (서버 동기화) ───

[MetricsForwarder] → 서버 전송 (적응형 3~30초 간격)
       ↓
[서버 응답: CViewSyncRecommendation]
       ↓
[validateAndComputeSyncSpeed()] → 서버 40% + 클라이언트 60% 블렌딩
       ↓
[PlayerViewModel.applySyncSpeed()] → 동일 경로로 적용
```

### 1.2 핵심 컴포넌트 파라미터 현황

| 컴포넌트 | 파라미터 | 현재 값 | 역할 |
|----------|----------|---------|------|

## 2. 버퍼링 발생 메커니즘 분석

### 2.1 버퍼링 유발 경로 식별

실시간 디버깅 및 코드 분석을 통해 **5가지 주요 버퍼링 유발 경로**를 식별하였습니다:

#### 경로 ① — PID 가속 → VLC 버퍼 고갈

```
높은 레이턴시(>6.8s) 감지
  → PID 출력: 양의 값 발생
  → 재생 속도 1.08x 적용
  → VLC가 세그먼트 소비 > 다운로드 속도
  → 내부 버퍼(500ms) 빠르게 소진
  → VLC 리버퍼링 (재생 중단)
  → 버퍼링 감지 → rate 1.0 리셋
  → 버퍼 회복 → 다시 가속 시도
  → ⚡ 반복 순환 (가속↔버퍼링 진동)
```

**위험도**: 🔴 높음  
**발생 조건**: lowLatency 프로파일(500ms 캐시)에서 네트워크 대역폭이 스트림 비트레이트의 ~1.2배 미만일 때

#### 경로 ② — 이중 속도 제어 충돌

```
LowLatencyController: rate 1.05 적용 (PID 기반)
  → 동시에 MetricsForwarder: 서버 추천 rate 0.98 적용
  → PlayerViewModel.applySyncSpeed() 덮어쓰기
  → 5초 후 다시 LowLatency: rate 1.06 적용
  → ⚡ 5초 주기 속도 진동 (1.05 → 0.98 → 1.06 → ...)
```

**위험도**: 🟡 중간  
**발생 조건**: 서버 동기화와 로컬 PID가 동시 활성화될 때 (방향 불일치 시 악화)

#### 경로 ③ — Seek 점프 후 버퍼 재구축 지연

```
레이턴시 >12s or 초기 진입 시 latency > target+2s
  → seek to target(6.0s) 실행
  → VLC 내부 버퍼 완전 플러시
  → 새 위치에서 세그먼트 다운로드 재시작
  → 500ms 캐시 재구축까지 ~2-3초 소요
  → ⚡ 눈에 보이는 끊김 + 검은 화면
```

**위험도**: 🟡 중간  
**발생 조건**: 첫 재생 진입 또는 네트워크 일시 불안정 후 회복 시

#### 경로 ④ — 멀티라이브 대역폭 경합 + 동시 가속

```
4개 스트림 동시 재생 (multiLive)
  → 스트림 A: rate 1.05 (뒤처짐 보정)
  → 스트림 B: rate 1.03 (뒤처짐 보정)
  → 총 대역폭 수요: Σ(bitrate_i × rate_i) > 가용 네트워크
  → BandwidthCoordinator: 2초 간격 갱신
  → 가속 반영 전 대역폭 계산으로 과대 할당
  → ⚡ 개별 스트림 버퍼링 연쇄 발생
```

**위험도**: 🟠 높음 (멀티라이브 한정)  
**발생 조건**: 4개 이상 동시 재생 + WiFi 혼잡 환경

#### 경로 ⑤ — PDT 측정 실패 → VLC fallback → 부정확한 레이턴시

```
PDT(EXT-X-PROGRAM-DATE-TIME) 파싱 실패
  → 3회 연속 범위 초과 → PDT 자동 중지
  → VLC 내부 버퍼 길이를 레이턴시로 사용
  → VLC buffer ≠ 실제 CDN-to-viewer 레이턴시
  → PID가 잘못된 측정값으로 속도 조정
  → ⚡ 불필요한 가속/감속 → 버퍼링 유발
```

**위험도**: 🟡 중간  
**발생 조건**: CDN 프록시에서 PDT 데이터 제거 또는 시계 오차가 큰 환경

## 3. 정량적 분석

### 3.1 버퍼 마진 대비 가속 소비 계산

**싱글 라이브 (lowLatency 프로파일)**:
- VLC 캐시: 500ms = 0.5초 분량의 세그먼트
- 최대 가속: 1.08x
- 가속 시 버퍼 소비 초과분: `0.08 × bitrate` (초당)
- 1080p 6Mbps 스트리밍 기준 초과 소비: 480kbps
- 500ms 버퍼 완전 소진 시간: `500ms / (0.08 × 소비율)` ≈ **6.25초**
- **즉, 가속 시작 후 ~6초면 버퍼 고갈 위험** (네트워크 여유 없을 시)

**멀티라이브 (multiLive 프로파일)**:
- VLC 캐시: 1500ms = 1.5초 분량
- 최대 가속: 1.08x
- 1500ms 버퍼 완전 소진 시간: ≈ **18.75초**
- 단, 4스트림 동시 가속 시 네트워크 경합으로 **실질 소진 시간 절반 이하**

### 3.2 PID 응답 시간 분석

```
PID 주기: 5.0초
EWMA α=0.15: 특정 레이턴시에 63% 수렴하는 데 필요한 샘플 수 ≈ 1/0.15 ≈ 6.7회
수렴 시간: 6.7 × 5.0s = ~33.5초

→ 갑작스러운 레이턴시 변화에 대한 PID 반응이 약 30초 이상 지체됨
→ 단, 이 보수적 설정이 오히려 과도한 보정을 방지하는 장점도 있음
```

### 3.3 버퍼 건강도 기반 클램핑 유효성

| 버퍼 건강도 (bh) | LowLatencyController 댐핑 | PlayerViewModel 최대 속도 | 실질 가속 |
|:-:|:-:|:-:|:-:|
| bh < 0.3 | 0.0 (차단) | 1.0 (차단) | 없음 |
| 0.3 ≤ bh < 0.6 | 0.5 (절반) | 1.02 | 최대 +2% |
| bh ≥ 0.6 | 1.0 (전체) | 1.08 | 최대 +8% |

**문제점**: 두 레이어의 클램핑 기준이 **동일 임계값**(0.3, 0.6)을 사용하여 이중 보호가 되지만, **실질적으로 bh가 0.6→0.3으로 급락하는 과도기**에서 LowLatency가 damping=0.5를 적용하는 동안 PlayerViewModel은 이미 max 1.02를 강제하므로, 실제 적용 속도는 `min(PID × 0.5, 1.02)` ≈ 1.01~1.02 수준. 이 구간에서의 추가 가속이 버퍼를 더 소진시킬 수 있음.

## 4. 발견된 문제 및 개선 방안

### 4.1 🔴 [Critical] PID Rate 진동 및 가속↔버퍼링 순환

**현상**: 가속(1.08x) → 버퍼 고갈 → rate 리셋(1.0x) → 버퍼 회복 → 다시 가속 → 반복

**근본 원인**:
- `pauseForBuffering()` 호출 시 rate를 1.0으로 즉시 리셋하지만, PID 내부 적분(I) 항이 리셋되지 않음
- 버퍼링 해제 후 `resumeFromBuffering()`이 호출되면 PID가 축적된 I 항 기반으로 즉시 높은 rate를 다시 출력
- 또한 `pauseForBuffering()`에서 EWMA와 latencyHistory를 리셋하지 않아 이전의 높은 레이턴시 기록이 PID에 잔류

**개선 방안**:
```
A. pauseForBuffering() 호출 시 PID 적분(I) 항 리셋 추가
   → pidController.reset() 또는 pidController.resetIntegral()

B. 버퍼링 해제 후 "쿨다운" 기간 도입
   → resumeFromBuffering() 후 10~15초간 최대 가속을 1.03으로 제한
   → 버퍼가 안정적으로 재구축될 시간 확보

C. 가속 이력 기반 진동 감지기 구현
   → 최근 60초 내 pauseForBuffering 호출 횟수 > 2 → 가속 상한을 1.03으로 자동 낮춤
   → 자동 복원: 120초간 버퍼링 없으면 정상 범위(1.08) 복귀
```

### 4.2 🟡 [High] 이중 속도 제어 경로 간 충돌 부재

**현상**: LowLatencyController(PID)와 MetricsForwarder(서버 추천)가 독립적으로 `applySyncSpeed()`를 호출하여 5초 주기로 속도가 뒤바뀔 수 있음

**근본 원인**:
- 두 경로 간 조율 메커니즘 없음
- 서버 추천은 네트워크 왕복 시간(RTT)만큼 지연된 데이터 기반
- PID는 로컬 실시간 데이터 기반 → 항상 더 최신

**개선 방안**:
```
A. 속도 결정 우선순위 도입
   → PID 활성 시: MetricsForwarder 속도 추천 무시 (PID가 최신 데이터 기반)
   → PID 비활성(데드존 내): MetricsForwarder 미세 조정 허용

B. 통합 속도 결정기 (Rate Arbiter) 도입
   → PID 출력과 서버 추천을 함께 받아 최종 속도 결정
   → 규칙: |PID output| > |server recommendation| → PID 사용, 그 외 서버 사용
   → 방향이 반대인 경우 보수적 수치(1.0에 가까운) 선택
```

### 4.3 🟡 [High] 버퍼 건강도(bufferHealth) 정의의 모호성

**현상**: `displayedPictures / decodedVideo` 비율이 실제 VLC 내부 버퍼 잔량과 다를 수 있음

**근본 원인**:
- `bufferHealth`는 VLC의 표시 프레임 vs 디코딩 프레임 비율로 계산됨
- 이는 "디코딩 파이프라인 효율"을 반영하지 "남은 버퍼 시간(초)"를 직접 나타내지 않음
- 버퍼가 500ms 남아있어도 프레임 비율이 0.95를 표시할 수 있고, 200ms에서도 0.90일 수 있음

**개선 방안**:
```
A. VLC의 실제 버퍼 잔량(초) 기반 건강도 계산
   → bufferHealth = (VLC 보고 캐시 잔량 ms) / (프로파일 liveCaching ms)
   → 직접적인 "몇 초 분량 남았는가" 정보 제공

B. 복합 건강도 지표 도입
   → 가중 평균: 0.6 × 시간 기반 건강도 + 0.4 × 프레임 비율 건강도
   → 두 지표가 일치하지 않을 때 보수적 값(낮은 쪽) 사용
```

### 4.4 🟡 [Medium] Seek 점프 후 버퍼 재구축 전략 부재

**현상**: maxLatency(12s) 초과 또는 초기 진입 seek 후 2-3초간 검은 화면/끊김

**개선 방안**:
```
A. Seek 후 preload grace 도입
   → onSeekRequired() 실행 후 자동으로 pauseForBuffering() 호출
   → 버퍼가 프로파일의 liveCaching 70% 이상 채워질 때까지 rate 1.0 유지
   → VLC의 재생 이벤트(.playing) 수신 후 2초 추가 대기

B. 점진적 접근 (Gradual Approach) — seek 대신 가속으로 수렴
   → latency > 8s 이상이되 12s 미만: seek 대신 1.08x 가속으로 서서히 수렴
   → latency > 12s: seek 실행 (불가피)
   → 이를 위해 seekRequired 판단 임계값을 3단계로 분리
```

### 4.5 🟡 [Medium] 멀티라이브 세션 간 가속 조율 미비

**현상**: 4개 세션이 각자 독립적으로 rate 조정 → 총 대역폭 예측 불가 → 동시 다발 버퍼링

**개선 방안**:
```
A. MultiLiveManager 레벨 가속 예산(Rate Budget) 도입
   → 전체 세션의 가속 합산이 임계값(예: 총 +12%) 이내로 제한
   → 우선순위: 선택된(포커스) 세션 > 비선택 세션
   → 비선택 세션은 최대 1.02 제한, 선택 세션이 나머지 예산 사용

B. BandwidthCoordinator 대역폭 계산에 playbackRate 반영
   → 현재: bandwidth = Σ(bitrate_i) / available_bandwidth
   → 개선: bandwidth = Σ(bitrate_i × rate_i) / available_bandwidth
   → ABR 화질 결정 시 가속에 의한 추가 수요 고려
```

### 4.6 🟢 [Low] 적응형 폴링 간격의 경계 효과 (Cliff Effect)

**현상**: MetricsForwarder의 500ms 데드존 경계에서 폴링 간격이 급변 (3s ↔ 30s)

**개선 방안**:
```
A. 히스테리시스 적용
   → 동기화→비동기화 전환: 데드존 + 200ms (700ms 초과 시)
   → 비동기화→동기화 전환: 데드존 − 100ms (400ms 이하 시)
   → 경계에서의 폴링 간격 진동 방지

B. 간격 변경 시 1단계씩 점진 전환
   → 현재: 상태에 따라 즉시 목표 간격으로 변경
   → 개선: 한 주기당 최대 1단계 변경 (30→20→12→8→5→3)
```

### 4.7 🟢 [Low] PDT 초기화 타임아웃 + Fallback 불안정

**현상**: PDT 측정이 6초 내 초기화 실패 시 VLC buffer fallback으로 전환되지만, 이후 PDT가 복구되어도 자동 복귀 불가

**개선 방안**:
```
A. PDT 폴링 중지 후 주기적 재시도 (60초마다)
   → 3회 연속 실패 → 중지 → 60초 후 5회 프로브 시도
   → 성공 시 PDT 모드 복귀
   
B. VLC fallback 모드일 때 PID 게인 20% 감소
   → 부정확한 측정 기반 과도한 보정 방지
```

## 7. 결론

### 7.1 현재 시스템 안정성 평가

| 항목 | 상태 | 비고 |
|------|------|------|
| 싱글 라이브 동기화 수렴 | ✅ 양호 | 보수적 PID로 30~40초 내 수렴 |
| 싱글 라이브 버퍼링 빈도 | ⚠️ 개선 필요 | 가속↔리셋 순환 가능성 존재 |
| 멀티라이브 동기화 | ⚠️ 주의 | 세션 간 가속 조율 미비 |
| 멀티라이브 버퍼링 빈도 | 🔴 위험 | 동시 가속 + 대역폭 경합 |
| PDT 측정 정확도 | ✅ 양호 | fallback 전환 메커니즘 구비 |
| Seek 안정성 | ⚠️ 개선 필요 | 재구축 지연 시 끊김 발생 |
| 서버 동기화 안정성 | ⚠️ 개선 필요 | PID와의 충돌 가능성 |

### 7.2 핵심 개선 요약

1. **즉시 적용**: PID 적분 리셋 + 쿨다운 기간 → 가속↔버퍼링 진동 해소
2. **단기 개선**: PID-서버 우선순위 분리 → 이중 제어 충돌 제거
3. **중기 개선**: 멀티라이브 가속 예산 + 점진적 수렴 → 근본적 안정성 확보
4. **장기 개선**: 버퍼 건강도 재정의 + 통합 속도 결정기 → 아키텍처 수준 정밀화

> Phase 1만 적용해도 체감 버퍼링 빈도를 **약 70~80%** 줄일 수 있을 것으로 예상됩니다.

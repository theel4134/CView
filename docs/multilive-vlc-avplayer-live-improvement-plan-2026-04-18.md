# 멀티라이브 내부 VLC / AVPlayer 엔진 라이브 스트리밍 재생 개선안

> 작성일: 2026-04-18  
> 대상: CView_v2 멀티라이브 재생 경로  
> 목표: **활성 세션 기준 1080p / 60fps / 8000kbps 유지** + 멀티세션 안정성 확보

---

## 1. 문서 목적

이 문서는 멀티라이브 화면에서 사용하는 **VLC 엔진**과 **AVPlayer 엔진**의 라이브 스트리밍 재생 품질을 개선하기 위한 내부 실행안을 정리한다.

핵심 목표는 다음 두 가지다.

1. **선택된 활성 세션은 1080p 60fps / 8000kbps 품질을 최대한 유지한다.**
2. **동시에 여러 세션이 재생되더라도 멈춤, 장기 버퍼링, 화질 진동을 최소화한다.**

> 전제: 멀티라이브에서 **모든 세션을 동시에 1080p60 / 8Mbps로 강제 유지**하는 것은 네트워크/디코딩/GPU 비용이 매우 크다. 따라서 본 문서는 **선택 세션 품질 고정 + 비선택 세션 자원 절감**을 기본 전략으로 한다.

---

## 2. 현재 코드베이스 진단 요약

현재 구현상 이미 좋은 기반이 존재한다.

### 2.1 이미 들어가 있는 장점

- **VLC 경로**
  - `forceHighestQuality = true` 시 해상도 캡핑, 프레임 스킵, hurry-up, skiploopfilter가 비활성화됨
  - `adaptive-logic=highest`로 최고 화질 유지 경로가 이미 존재함
- **AVPlayer 경로**
  - `isQualityLocked = true`
  - `lockedPeakBitRate = 8_000_000`
  - `lockedMaximumResolution = 1920×1080`
- **멀티라이브 공용 제어**
  - `MultiLiveBandwidthCoordinator`가 세션별 대역폭 배분과 버퍼 상태를 관리함
  - `MultiLiveEnginePool`로 엔진 재사용 경로가 있어 생성 비용이 줄어듦

### 2.2 현재 남아 있는 핵심 문제

#### A. 활성 세션과 비활성 세션의 정책이 충분히 분리되지 않음

현재는 멀티라이브에서 선택 세션도 `multiLive` 성향의 정책 일부를 공유한다. 품질 잠금이 있어도 버퍼 정책, 갱신 주기, 선택/비선택 승격 흐름이 완전히 독립적이지 않다.

#### B. 품질 유지와 안정성 유지가 서로 다른 계층에서 동시에 제어됨

- StreamCoordinator
- LowLatencyController
- ABRController
- MultiLiveBandwidthCoordinator
- 각 엔진 내부 옵션

이 경로들이 겹치면 **품질 하향/복귀 진동**, **불필요한 재연결**, **버퍼 오판**이 발생할 수 있다.

#### C. VLC와 AVPlayer의 역할 분담이 문서화되어 있지 않음

코드상 두 엔진 모두 멀티라이브에 사용 가능하지만, 어떤 상황에서 어느 엔진을 우선해야 하는지 기준이 약하다.

## 3. 목표 아키텍처

### 3.1 세션 계층 분리

멀티라이브 세션은 아래 3단계로 명확히 분리한다.

| 계층 | 상태 | 목표 품질 | 목적 |
| ---- | ---- | --------- | ---- |
| Tier 1 | 선택된 활성 세션 | **1080p60 / 8Mbps 고정** | 메인 시청 품질 보장 |
| Tier 2 | 화면에 보이는 비선택 세션 | 480p~720p 적응 | 전체 안정성 유지 |
| Tier 3 | 가려진/비가시 세션 | 비디오 중지 또는 최소화 | CPU/GPU/네트워크 절감 |

### 3.2 원칙

- **활성 세션 1개만 최고 품질을 강하게 보장**한다.
- 비선택 세션은 품질보다 **즉시성, 안정성, 리소스 효율**을 우선한다.
- 엔진 전환은 빈번하게 하지 않고, **세션 승격/강등 시점**에서만 제한적으로 수행한다.

## 4. 엔진별 개선안

### 4.1 VLC 엔진 개선안

#### VLC 권장 방향

VLC는 멀티라이브에서 다음 상황에 강하다.

- 고급 영상 옵션/이퀄라이저/오디오 필터가 필요한 경우
- 기존 VLC 기반 진단/메트릭 경로를 활용해야 하는 경우
- PiP 또는 고급 도구 탭 연동이 중요한 경우

### VLC 개선 포인트

#### 1) `multiLiveHQ` 전용 프로파일 추가

현재 `multiLive` 프로파일과 `forceHighestQuality` 조합으로도 품질 잠금은 가능하지만, **선택 세션 전용의 버퍼/갱신/디코더 정책**이 필요하다.

권장값:

| 항목 | 현재 멀티라이브 성향 | 제안값 (선택 세션) |
| ---- | -------------------- | ------------------ |
| network/live caching | 1500ms | **700~900ms** |
| manifest refresh | 20초 계열 | **4~6초** |
| decoder threads | 1 중심 | **2~4** |
| adaptive logic | highest / predictive 혼합 | **항상 highest** |
| skip frames / hurry-up | 비선택 최적화용 사용 | **선택 세션 비활성화 유지** |

#### 2) 선택 세션 승격 시 엔진 재생성 최소화

선택 전환 시 전체 재생 파이프라인을 다시 만드는 대신,

- 기존 drawable 유지
- variant URL만 교체
- 통계 타이머만 승격
- manifest refresh 주기만 상향

으로 처리하면 검은 화면과 초기 버퍼링을 줄일 수 있다.

#### 3) VLC는 “선택 세션 전용 HQ 경로”와 “비선택 세션 절약 경로”를 코드에서 명확히 분기

즉, 아래처럼 분리해야 한다.

- 선택 세션: 화질 잠금 + 짧은 캐시 + 낮은 레이턴시
- 비선택 세션: 해상도 캡핑 + 긴 캐시 + skip/hurry-up 허용

---

### 4.2 AVPlayer 엔진 개선안

#### AVPlayer 권장 방향

멀티라이브의 **주력 메인 시청 엔진은 AVPlayer를 우선 후보**로 두는 것이 바람직하다.

이유:

- macOS VideoToolbox 경로가 비교적 안정적임
- 1080p/8Mbps 상한을 코드에서 직접 잠글 수 있음
- 선택 세션의 장시간 재생에서 vout 재바인딩 부담이 VLC보다 적음

### AVPlayer 개선 포인트

#### 1) 선택 세션에서는 AVPlayer 화질 잠금을 유지하되, 멀티라이브 예외 로직을 단순화

현재는 멀티라이브 + AVPlayer 조합에서 direct variant pinning을 일부 회피하고 있다. 이 전략은 유지하되, 다음 원칙으로 단순화한다.

- AVPlayer는 **마스터 플레이리스트 기반 ABR**은 유지
- 단, 상한은 **1920×1080 / 8Mbps**로 강하게 제한
- 멀티라이브 선택 세션에서만 `isQualityLocked = true`
- 비선택 세션은 lock 해제 또는 bitrate cap 하향

#### 2) 버퍼 정책 이원화

AVPlayer도 활성/비활성 세션 버퍼 목표를 나눠야 한다.

| 세션 상태 | 버퍼 전략 |
| --------- | --------- |
| 활성 세션 | **1.0~1.5초 전방 버퍼**, 빠른 복구 중심 |
| 비선택 세션 | **3~5초 버퍼**, 안정성 중심 |

이렇게 하면 선택 세션은 레이턴시를 줄이고, 비선택 세션은 끊김을 방지할 수 있다.

#### 3) 선택 세션 승격 시 워밍업 적용

활성 세션으로 승격되기 직전,

- master playlist refresh
- 현재 네트워크 여유 측정
- 첫 세그먼트 미리 당겨오기

를 수행하면 1080p60 진입 시 첫 프레임 지연을 줄일 수 있다.

## 5. 공통 개선안

### 5.1 “품질 유지 계층”을 하나로 통합

현재는 품질 관련 의사결정이 여러 레이어에 흩어져 있다. 아래 순서로 우선권을 고정해야 한다.

1. **사용자 강제 품질 잠금**
2. **선택 세션 HQ 정책**
3. **엔진 내부 ABR/적응 품질**
4. **긴급 강등 (정말 필요한 경우만)**

즉, 활성 세션이 HQ 잠금 상태라면:

- 일반 ABR downgrade 무시
- 대역폭 코디네이터도 활성 세션은 보호
- 긴급 강등은 전체 네트워크 붕괴 시에만 허용

### 5.2 대역폭 배분 정책 재정의

권장 규칙:

- 활성 세션은 최소 **8Mbps 보장 대상**으로 예약
- 남는 대역폭을 비선택 세션이 분배 사용
- 총 가용 대역폭이 부족하면 **비선택 세션부터 먼저 하향**
- 선택 세션은 마지막까지 보호

예시:

| 총 가용 대역폭 | 선택 세션 | 나머지 3세션 |
| ------------- | --------- | ------------- |
| 30Mbps 이상 | 1080p60 / 8Mbps | 720p~1080p |
| 18~24Mbps | 1080p60 / 8Mbps 유지 | 480p~720p |
| 12~16Mbps | 1080p60 / 8Mbps 우선 유지 | 360p~480p |
| 10Mbps 이하 | 필요 시 선택 세션만 유지 | 나머지 정지/숨김 |

### 5.3 선택 전환 시점의 사전 준비

선택 세션을 바꾸는 순간 바로 고화질 전환하면 튀는 경우가 많다. 따라서 다음 3단계가 필요하다.

1. 새 선택 대상 세션의 manifest/variant 선갱신
2. 1~2초간 warm-up 상태 유지
3. 이후 1080p60 / 8Mbps lock 활성화

이 방식은 “선택 즉시 검은 화면 → 뒤늦게 선명해짐” 현상을 줄인다.

### 5.4 메트릭 수집 주기 차등화

모든 세션에서 고빈도 metrics / latency / stat timer를 유지하면 CPU 오버헤드가 커진다.

권장값:

| 세션 상태 | 메트릭 수집 주기 |
| --------- | ---------------- |
| 선택 세션 | 1~2초 |
| 비선택 세션 | 5초 |
| hidden | 10초 또는 중단 |

## 6. 내부 테스트 기준

### 6.1 필수 성공 기준

#### 선택 세션 기준

- 평균 해상도: **1920×1080 유지**
- 체감 프레임레이트: **55~60fps 유지**
- 평균 비트레이트: **7.2~8.0Mbps 이상**
- 10분 재생 중 장기 버퍼링: **1회 이하**
- 선택 전환 후 첫 화면 복귀: **2초 이내**

#### 멀티라이브 전체 기준

- 4세션 동시 재생 시 앱 전체가 멈추지 않을 것
- 비선택 세션 강등은 허용하되, 선택 세션 품질은 흔들리지 않을 것
- 선택 전환을 반복해도 drawable/vout deadlock이 재현되지 않을 것

### 6.2 관찰 지표

- dropped frames
- decoded fps
- actual displayed resolution
- average bitrate
- rebuffer count
- focus switch latency
- CPU / GPU 사용량
- memory growth during 30min session

## 7. 실행 우선순위

### P0 — 바로 적용할 항목

1. **선택 세션용 `multiLiveHQ` 프로파일 신설**
2. **선택 세션은 HQ 잠금 우선, 비선택 세션은 우선 강등**
3. **VLC/AVPlayer 모두 활성/비활성 세션 버퍼 정책 분리**
4. **선택 전환 시 warm-up → lock 적용의 2단계 승격 도입**

### P1 — 안정화 항목

1. 엔진별 기본 우선순위 정리
   - 기본: AVPlayer
   - 기능 우선/특수 채널: VLC
2. metrics/latency 수집 주기 차등화
3. 긴급 강등 규칙을 선택 세션 보호형으로 재구성

### P2 — 검증/운영 항목

1. 4세션 / 30분 장시간 soak test
2. Wi-Fi 혼잡 / 유선 / 저대역폭 환경별 비교표 작성
3. 엔진별 회귀 테스트 시나리오 정립

## 9. 결론

현재 코드베이스는 이미 **1080p / 8Mbps 품질 잠금 기반**을 어느 정도 갖추고 있다. 따라서 새로 완전히 갈아엎기보다는,

- **선택 세션 전용 HQ 정책을 분리**하고,
- **비선택 세션 자원 절감 전략을 더 강하게 적용**하며,
- **VLC와 AVPlayer의 역할을 명확히 구분**하는 방식이 가장 현실적이다.

가장 중요한 판단은 하나다.

> **멀티라이브에서 모든 세션의 최고 품질을 동시에 고집하지 말고, 선택 세션 1개에 품질 예산을 집중해야 한다.**

이 원칙을 유지하면, 내부 테스트 기준인 **1080p 60fps / 8000kbps 유지**와 **멀티라이브 안정성**을 동시에 만족시킬 가능성이 가장 높다.
